# Agrirent Implemented Features
## T-01: FR-Farmer/Admin/Employee-User Registration
Screen Definition: Email & Password, Name, Phone Number field, Signup Button, Third Party SSO
User Story: As a user, I want to create an account so I can access the platform.
Acceptance Criteria: The system validates email, password, re-password, and user type before signup.
The system stores user credentials with inactive status and sends an email verification link.
The system activates the account only after valid email verification and restricts login before verification.
The system allows users to resend verification email and enforces link expiration time.
## T-21: FR-Employee- Schedule Equipment Maintenance 
Screen Definition: Equipment List, Maintenance Date, Status (Scheduled/In Maintenance/Completed), Save Button  User Story: As an employee, I want to schedule maintenance so that equipment remains usable and safe.  Acceptance Criteria:  1. The system should allow employees to assign maintenance dates. If maintenance scheduling fails or not completed, the system does not assign any date and displays an error message: "Maintenance scheduling failed."  2. The system should restrict booking for equipment under maintenance. If maintenance is not active or not scheduled, the system allows the equipment to be booked.  3. The system should display and updates maintenance status when maintenance is scheduled or in progress. If maintenance information is unavailable or not set, the system does not display any maintenance status. The system saves maintenance history after maintenance activities occur. If no maintenance activity is recorded or saving fails, the system does not store any history.  4. The system should display a success message after maintenance is successfully scheduled. If scheduling fails or is incomplete, the system displays an error message: "Maintenance scheduling failed."  Status: Implemented 
## T-22: FR-Employee- Inspect Returned Equipment
Screen Definition:Return List, Condition Options (Good/Damaged/Needs Service), Comment Box, Submit Button
User Story: As an employee, I want to inspect returned equipment so I can determine its condition.
The system should allow employees to select the condition after return, and if the condition is not selected or the process fails, the system does not record the condition and displays an error message: "Condition selection failed." Additionally, the system should allow users to add additional notes in the comment field, but if the comment field is not filled or fails to save, the system does not store any additional notes. Furthermore, the system should mark damaged equipment as unavailable, whereas if the equipment is not marked as damaged, the system keeps it available for use. The system should also store inspection data after the return process, and if inspection data is not recorded or saving fails, the system does not store any inspection information. Finally, the system should display a confirmation message after successful submission, and if the process fails or is incomplete, the system displays an error message: "Confirmation failed."
## T-23: Fr-Farmer-Request Rental Cancellation
Screen Definition: Booking List, Cancel Button, Reason Input, Submit Button 
User Story: As a farmer, I want to cancel my booking so I can adjust my plans.
The system should allow cancellation only before delivery, and if the delivery is already completed or in progress, the system does not allow cancellation and displays an error message: "Cancellation not allowed after delivery." Additionally, the system should require the user to provide a reason for cancellation, but if no reason is provided or submission fails, the system does not process the cancellation and displays an error message: "Cancellation reason required." Furthermore, the system should update the booking status to "Cancelled" after a successful cancellation, whereas if the cancellation process fails, the system does not update the booking status. The system should also send a cancellation notification to the admin after a successful cancellation, and if notification delivery fails, the admin does not receive any alert. Finally, the system should keep cancelled bookings in the history, but if the saving process fails, the system does not store the cancelled booking in history.
## T-24: FR-Farmer-Submit Equipment Review & Rating
Screen Definition: Rating Stars (1–5), Review Text Box, Submit Button
User Story: As a farmer, I want to rate equipment so I can share my experience.
Acceptance Criteria: The system should allow the user to select a rating between 1 to 5 stars, and if no rating is selected or the process fails, the system does not submit the rating and displays an error message: "Rating selection required." Additionally, the system should allow the user to write a review in the text box, but if the review field is empty or the submission fails, the system does not submit the review and displays an error message: "Review description cannot be empty." Furthermore, the system should store the rating and review after successful submission, whereas if the saving process fails, the system does not store the review and displays an error message: "Review submission failed." Finally, the system should display a confirmation message after successful submission, and if the process is completed, the system displays a message: "Review submission successful."
## T-25: FR-User- Manage Complaint & Support Tickets
Screen Definition: Complaint Form, Category Dropdown, Description Box, Submit Button.
User Story: As a user, I want to report issues so they can be resolved quickly.
Acceptance Criteria: The system should allow the user to select a complaint category, and if no category is selected or the process fails, the system does not submit the complaint and displays an error message: "Complaint category selection failed." Additionally, the system should require the description field to be filled before submission, but if the description is empty or the submission fails, the system does not submit the complaint and displays an error message: "Description cannot be empty." Furthermore, the system should generate a unique ticket ID after successful complaint submission, whereas if ticket generation fails, the system does not create the complaint record and displays an error message: "Ticket ID generation failed." The system should also allow the admin to update the complaint status, and if the update process fails or is incomplete, the system does not change the complaint status and displays an error message: "Complaint status update failed." Finally, the system should display complaint status updates to the user, but if the status is not updated or cannot be retrieved, the system does not show any new complaint status information.
